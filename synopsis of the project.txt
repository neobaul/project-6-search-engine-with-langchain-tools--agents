## Layer 3 — The Callback Handler

```python
class UIThreadCallbackHandler(BaseCallbackHandler):
    def on_agent_action(self, action, **kwargs):
        self.emitter.emit(f"[Agent Action]: {action.tool} -> Input: {action.tool_input}")
    def on_tool_end(self, output, **kwargs):
        self.emitter.emit(f"[Tool Finished]...")
    def on_llm_start(self, serialized, prompts, **kwargs):
        self.emitter.emit(f"[LLM Prompting]...")
```

This is a **spy that watches the agent's brain in real time**. Every time the agent:
- decides to use a tool → `on_agent_action` fires
- gets a result back from a tool → `on_tool_end` fires
- sends a prompt to the LLM → `on_llm_start` fires

Each event emits a signal that gets printed to your green log console at the bottom of the UI. That's how you see the agent "thinking" live.

## Layer 5 — The AgentWorker (The Engine Room)

```python
class AgentWorker(QThread):
    output_signal = pyqtSignal(str)
    log_signal = pyqtSignal(str)
    finished_signal = pyqtSignal()
```

This is the **most important class**. It inherits from `QThread`, meaning it runs in a **separate background thread**. This is crucial — if the AI ran in the main thread, your UI would completely freeze while it thinks.

The three signals are how it communicates back to the UI:
- `output_signal` → sends the final answer to the chat display
- `log_signal` → sends trace messages to the green log console
- `finished_signal` → tells the UI "I'm done, re-enable the button"

This is the **routing logic** — a really smart design decision. Two different agent architectures are used:

- **ReAct** (for Groq/Ollama) — the agent reasons in a loop: *Thought → Action → Observation → Thought...* It's more compatible with open-source models that don't natively support tool calling
- **Tool-Calling** (for OpenAI/Anthropic) — these models natively understand structured tool calls, so a more direct architecture is used

`AgentExecutor` is the **loop runner** — it keeps cycling the agent through Thought/Action/Observation until the agent decides it has a final answer. The callback handler rides along and reports everything to your log console.

## The Flow Summary — End to End

```
You type a question → execute_search() fires
→ AgentWorker spins up in background thread
→ Agent gets Tavily + WebScraper tools
→ Agent thinks: "I should search for X" → calls Tavily
→ Callback handler logs "[Agent Action]: TavilySearch"
→ Agent gets results, maybe scrapes a URL deeper
→ Agent synthesizes final answer
→ output_signal fires → answer appears in chat
→ Q&A pair stored in ChromaDB
→ finished_signal fires → button re-enables
```
